# AUTONOMOUS WORKER PROTOCOL

[ЭТАП] `AS-CODEX-AUTONOMOUS-WORKER-001`

## [ЦЕЛЬ]

GitHub является асинхронным каналом задач и отчётов. Пользователь не переносит рабочие сообщения между ChatGPT и Codex вручную.

## [РОЛИ]

- ChatGPT читает репозиторий, анализирует отчёты и формирует следующую безопасную задачу.
- Codex worker периодически синхронизирует `main`, читает `00_CONTROL/ACTIVE_TASK.md`, выполняет новую разрешённую задачу и пишет результат в GitHub.
- Пользователь нужен только для выдачи недостающего доступа и явного подтверждения опасного действия.

## [КАНОНИЧЕСКИЕ ФАЙЛЫ]

- `00_CONTROL/ACTIVE_TASK.md` — единственный исполняемый источник задачи.
- `00_CONTROL/TO_CODEX.md` — человекочитаемая копия/очередь сообщения для Codex; сама по себе не исполняется.
- `00_CONTROL/CODEX_REPORT.md` — полный результат последней работы.
- `00_CONTROL/DECISION_REQUIRED.md` — решения и доступы, без которых безопасное продолжение невозможно.
- `00_CONTROL/TO_CHATGPT.md` — короткий ответ для чтения ChatGPT.
- `00_CONTROL/AUTONOMOUS_WORKER_STATUS.md` — состояние протокола и последнего проверенного запуска.

## [ФОРМАТ НОВОЙ ЗАДАЧИ]

Worker запускает Codex только если `ACTIVE_TASK.md`:

1. содержит точную метку `[CHATGPT→CODEX]`;
2. не содержит статус `выполнено`, `completed` или `done`;
3. отличается по SHA-256 от последней обработанной версии;
4. получен после успешного `git pull --ff-only`;
5. рабочий checkout не содержит незавершённых изменений.

Минимальный рекомендуемый формат:

```text
[CHATGPT→CODEX]
[ЭТАП]
TASK-ID

[СТАТУС]
READY

[ЗАДАЧА]
...

[ЗАПРЕТ]
...
```

## [ЦИКЛ WORKER]

1. Захватить локальный lock, исключающий второй экземпляр.
2. Создать или открыть отдельный checkout `.runtime/repo`.
3. Проверить чистоту checkout.
4. Выполнить `git pull --ff-only origin main`.
5. Прочитать и проверить `ACTIVE_TASK.md`.
6. Запустить `codex exec` в `workspace-write`, non-interactive и ephemeral режиме.
7. Если требуется новый approval, deploy, секрет, запись в production, массовое или разрушительное действие — не выполнять его; записать блокер.
8. Проверить изменённые пути и выполнить локальный поиск секретов без вывода значений.
9. Создать commit и push только после успешных проверок.
10. Сохранить локальный hash/status, чтобы не выполнять одну задачу повторно.

## [РАЗРЕШЁННЫЕ ПУТИ]

Worker может коммитить только проектные пути:

`00_CONTROL/`, `00_START_HERE/`, `01_CONTEXT/`, `02_STABLE_DATA/`, `06_REPORTS/`, `07_CURRENT_TASKS/`, `10_ARCHITECTURE/`, `tools/`, `.gitignore`, `README.md`, `AGENTS.md`.

Любое изменение вне списка останавливает push.

## [ОПАСНЫЕ ДЕЙСТВИЯ]

Без отдельного явного подтверждения пользователя worker не выполняет:

- deploy/publish и изменение внешних систем;
- запись в production Apps Script, n8n, Telegram, Google Sheets или формы;
- массовые изменения данных;
- работу с токенами, ключами, cookies и credentials;
- разрушительные Git/файловые операции;
- расширение sandbox или обход approvals;
- отправку сообщений людям или создание платных ресурсов.

Non-interactive Codex не может показать новый approval. Такое действие должно завершиться отчётом и записью в `DECISION_REQUIRED.md`.

## [ОГРАНИЧЕНИЕ КАНАЛА]

[БЛОКЕР] Локальный worker не может получить новую задачу от ChatGPT, пока ChatGPT имеет только read-доступ к GitHub. Для режима без ручного переноса нужен один writable inbox: право обновлять `ACTIVE_TASK.md` либо другой отдельно утверждённый GitHub-канал с записью.

## [ОДИН СЛЕДУЮЩИЙ ШАГ]

Выдать ChatGPT минимальное право записи только в task-inbox или утвердить эквивалентный writable канал; не расширять доступ к секретам и production.
