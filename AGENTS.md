# Академия Стратег — правила работы Codex

## Главная цель проекта

Главная цель проекта «Академия Стратег» — построить масштабируемую систему продвижения, продаж, обучения продавцов / партнёров / Бэтманов и привлечения собственников бизнеса к технологии стратегического управления.

Во главе проекта стоит не таблица, не бот, не amoCRM, не n8n, не Cursor, не Google Form, не Zoom и не отдельная ссылка.

Все ресурсы, сервисы, таблицы, workflow, формы, боты, скрипты и файлы — только инструменты для достижения цели.

## Основная воронка

источник / контент / рекомендация / стажёр  
→ собственник бизнеса  
→ бизнес-тест  
→ анализ / разбор  
→ продажа обзорного курса стратегического управления  
→ дальнейшая линейка Академии.

## Перед любым действием проверять

1. Как это ведёт к главной цели?
2. Какой участок воронки усиливает?
3. Какой конкретный результат даст?
4. Что может сломаться?
5. Это временная правка или часть целевой архитектуры?
6. Как это масштабируется?
7. Как потом встроится в amoCRM / n8n / Cursor / Telegram / Zoom / Calendar?
8. Что подтверждено данными?
9. Что является предположением?
10. Какой один следующий шаг?

## Жёсткие запреты

Нельзя переписывать рабочие workflow без отдельного решения.

Нельзя менять названия листов, колонок, credentials, webhook URL, Telegram credentials, Google Sheets mappings без явного разрешения.

Нельзя импортировать чужие workflow без ручной сверки.

Нельзя начинать внедрение, если не прочитан стабильный контекст проекта.

Нельзя чинить локальную ошибку так, чтобы ломалась вся цепочка.

Нельзя создавать новую версию рабочего процесса вместо исправления подтверждённого рабочего слоя без отдельного решения.

Нельзя класть в проект реальные токены, пароли, API-ключи, cookies и credentials.

## API / Secret / Token handling

Перед любой задачей с n8n, webhook, API, Google Sheets, Telegram, OAuth, HMAC или credentials сначала определить тип доступа:

1. n8n API key — только для управления n8n, программистам сайта не передавать.
2. HMAC secret — только для подписи webhook-запросов, передавать программистам отдельно безопасным каналом.
3. Google Sheets credential — только для n8n, наружу не передавать.
4. Telegram token — только для бота/n8n, наружу не передавать.
5. OAuth/service account — внутренний доступ, наружу не передавать.

Перед deploy/fix обязательно проверить:

- Codex видит n8n API;
- Codex видит live workflow;
- Codex видит executions;
- создан backup live workflow;
- понятно, какой ключ нужен;
- понятно, где он хранится;
- понятно, кому его можно передавать;
- понятно, что нельзя писать в чат/отчёты/JSON.

Все реальные секреты хранить только вне проекта:

C:\Users\admin\.secrets\

В проектных JSON, отчётах и backup-файлах использовать только:

[REDACTED_INLINE_SECRET]  
[REDACTED_API_KEY]  
[REDACTED_TOKEN]

Запрещено:

- печатать secret/API key в консоль;
- писать secret/API key в отчёты;
- писать secret/API key в JSON workflow;
- отправлять secret/API key в ChatGPT;
- передавать n8n API key программистам;
- передавать Google Sheets credentials программистам;
- передавать secret в URL/query/body;
- делать скрины с видимыми secret/API key.

Если нужно передать secret человеку:

- скопировать в Windows clipboard;
- вывести только:
  SECRET COPIED TO CLIPBOARD
  length: <n>
- не показывать значение.

Если задача может быть сделана через API — делать через API, не гонять Максима по кнопкам.

Если API недоступен — сначала остановиться и подтвердить причину, а не патчить вслепую.

## Режим технической работы

Работать по схеме:

один шаг → пользователь выполняет → пользователь присылает скрин / execution / файл → проверить → только потом следующий шаг.

Не давать 10 действий сразу.

Если данных не хватает — запросить только недостающие данные.

## Стабильные данные

Все подтверждённые решения фиксировать в формате:

[ЗАПРЕТ]  
[ГИПОТЕЗА]
