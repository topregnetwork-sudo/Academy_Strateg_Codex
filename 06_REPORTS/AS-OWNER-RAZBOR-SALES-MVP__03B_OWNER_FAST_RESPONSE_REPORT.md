## [РИСКИ]

- Оставшаяся задержка формируется в обязательном критическом пути:
  чтение владельца, `Сообщения_OWNER`, квалификации и первый запуск Code resolver в n8n Cloud.
- Первый Code node добавляет около **1,7 с** на execution.
- Последовательные Google Sheets reads добавляют ещё примерно **1,7–2,7 с** в зависимости от запроса.
- Текущий Google Sheets OAuth credential технически запрещён для HTTP Request, поэтому Google API `batchGet` с этой учётной записью невозможен.
- Вариант переноса resolver в штатное expression-поле несовместим с синтаксисом n8n для многострочного JS; он снят.
- Без отдельного архитектурного решения дальнейшее редактирование текущего рабочего слоя создаёт риск повторного отсутствия ответа собственнику.

## [РЕШЕНИЕ, КОТОРОЕ НУЖНО ПРИНЯТЬ]

Рекомендуемый вариант: создать внутри n8n быстрый кэш только для критических OWNER-данных, синхронизируемый из Google Sheets:

- `Сообщения_OWNER` остаётся источником обычных клиентских текстов;
- владельцы и состояние квалификации остаются источником бизнес-состояния;
- критический путь читает их локальные копии из n8n Data Tables;
- запись ответа в `Квалификация` остаётся до клиентского ответа;
- Google Sheets-логи и зеркала остаются асинхронными.

Это отдельное архитектурное изменение и не было разрешено текущим заданием, поэтому не внедрялось без подтверждения.

Альтернатива: отдельный Google OAuth credential, разрешённый для HTTP Request, и один Google Sheets API `batchGet`. Текущий credential для этого непригоден.
