# AS-OWNER-RAZBOR-MVP__01_POST_TEST_BOT_BOOKING

## [ПОДТВЕРЖДЕНО]

- Стабильный контекст проекта принят.
- Текущий webhook `AS-BTM-OWNER-TEST-MVP__03_WEBHOOK_REGISTRATION_AND_RESULT` не трогаем: он уже принимает `business_test_registration` и `business_test_result`.
- Следующий слой начинается после получения результата бизнес-теста: результат теста -> приглашение в Telegram-бот -> квалификация -> выбор времени разбора -> напоминания -> статус разбора -> подготовка к продаже курса.
- В локальных workflow нет отдельного owner/razbor workflow. Найдены только текущие HR/Batman/test workflow.
- В подтверждённом снимке внешней таблицы есть листы `БизнесТест_Регистрации`, `БизнесТест_Результаты`, `Логи_Webhook_БизнесТест`, но нет owner-слоя разборов.

## [НОВАЯ ВОРОНКА]

1. Бэтман выдаёт владельцу бизнеса ссылку на `business_test_main`.
2. Владелец проходит бизнес-тест.
3. Сайт отправляет результат в текущий n8n webhook.
4. Сайт показывает страницу результата.
5. На странице результата появляется кнопка `Получить разбор результата`.
6. Кнопка ведёт в отдельный Telegram-бот для владельцев через deep-link.
7. Бот распознаёт связанный результат теста по безопасному токену.
8. Бот задаёт короткую квалификацию.
9. Бот показывает свободные слоты разбора.
10. Владелец выбирает слот.
11. Система фиксирует запись и блокирует слот.
12. Система отправляет напоминания.
13. После разбора фиксируется статус: проведён / не пришёл / перенесён.
14. Следующий шаг: продажа обзорного курса стратегического управления.

## [БОТ]

Рекомендуется отдельный Telegram-бот для владельцев бизнеса и разборов.

Не рекомендуется использовать HR/Batman bot автоматически, потому что:

- владельцы бизнеса не должны видеть логику Бэтманов;
- команды `Бэтмен`, `Команда`, `Моя статистика` относятся к другому контуру;
- данные телефонов, результатов теста и записи на разбор должны быть отделены от HR/Batman слоя;
- смешивание увеличит риск случайной выдачи ссылок, неправильных команд и утечки данных.

Рабочее имя контура: `AS-OWNER-BOT`.

## [DEEP-LINK]

Формат будущей ссылки:

```text
https://t.me/<OWNER_BOT_USERNAME>?start=r_<secure_token>
```

Токен должен быть непрозрачным и безопасным. В Telegram URL нельзя класть:

- телефон;
- email;
- ФИО;
- полный результат теста;
- webhook auth token;
- персональные данные;
- scores/syndromes JSON.

Что должен связывать токен на стороне системы:

- `registration_id`;
- `result_id`;
- `test_session_id`;
- `btm_id`;
- `result_url`;
- телефон/email владельца в таблице, но не в ссылке;
- `created_at`;
- `expires_at`;
- `token_status`.

Для MVP допустим непрозрачный UUID/рандомный токен. Целевой безопасный вариант: хранить `token_hash`, а не сырой токен.

## [КВАЛИФИКАЦИЯ]

MVP-квалификация должна быть короткой:

1. Сколько сотрудников в компании?
2. Примерная выручка?
3. Город?
4. Кто проходил тест: собственник / руководитель / сотрудник / другое?

Длинную анкету не вводить без отдельного решения Максима.

## [СЛОТЫ]

Рассмотрены 3 варианта.

Вариант A: Google Calendar как источник свободных слотов.

- Плюс: ближе к реальному календарю.
- Минус: больше интеграционных рисков, сложнее защита от дублей, нужны Calendar credentials и правила занятости.

Вариант B: Google Sheet `Разборы_Слоты` как источник слотов.

- Плюс: быстрее всего для MVP, соответствует текущей архитектуре Google Sheets + n8n.
- Плюс: проще аудитить и сверять руками.
- Минус: нужна аккуратная защита от double-booking.

Вариант C: смешанный слой: Sheet хранит правила/экспертов/служебные статусы, Calendar хранит занятость.

- Плюс: лучший целевой вариант.
- Минус: не лучший первый шаг, пока нет подтверждённого owner bot и правил разбора.

Рекомендация для MVP: начать с варианта B, но с обязательной защитой от двойной записи:

- слот выбирается только если `статус = свободен`;
- update идёт по `slot_id`;
- перед записью workflow повторно перечитывает слот;
- если слот уже не свободен, бот предлагает другой слот;
- после записи слот получает `статус = забронирован`, `owner_id`, `booking_id`, `locked_at`.

## [WORKFLOWS]

Проектировать, но пока не импортировать:

1. `AS-OWNER-BOT__00_MAIN_ROUTER`
   - принимает `/start r_<secure_token>`;
   - валидирует токен;
   - связывает Telegram chat с owner/result;
   - отправляет владельца в квалификацию.

2. `AS-OWNER-RAZBOR__01_QUALIFICATION`
   - задаёт 4 MVP-вопроса;
   - пишет ответ в `Владельцы_Квалификация`;
   - переводит владельца к выбору слота.

3. `AS-OWNER-RAZBOR__02_SLOT_SELECTION`
   - показывает свободные слоты;
   - повторно проверяет слот перед записью;
   - создаёт `booking_id`;
   - обновляет `Разборы_Слоты` и `Разборы_Записи`.

4. `AS-OWNER-RAZBOR__03_REMINDERS`
   - отправляет напоминания по будущим записям;
   - пишет статусы в `Разборы_Напоминания` и `Разборы_Записи`.

5. `AS-OWNER-RAZBOR__04_STATUS_UPDATE`
   - фиксирует итог разбора: проведён / не пришёл / перенесён;
   - готовит следующий шаг к продаже курса.

## [РИСКИ]

- Double-booking: два владельца могут выбрать один слот, если workflow не перечитывает слот перед update.
- Смешивание контуров: owner-бот нельзя сливать с HR/Batman командами без отдельного решения.
- Утечка персональных данных: нельзя кодировать телефон/email/ФИО/результат теста в Telegram deep-link.
- Потеря связки результата: если сайт не сохранит `test_session_id`/`result_id`, бот не поймёт, какой результат разбирать.
- Calendar рано подключать: без правил экспертов и занятости появится лишняя сложность.
- Google Sheets как база MVP требует дисциплины статусов и `slot_id`.
- Cached schema n8n может не увидеть новые колонки до ручной сверки.
