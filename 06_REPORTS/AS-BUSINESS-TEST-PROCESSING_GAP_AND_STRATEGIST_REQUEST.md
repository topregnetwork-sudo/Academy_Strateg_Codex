## [ЧТО УЖЕ ОБРАБАТЫВАЕТСЯ]

n8n webhook сейчас делает:

- проверку token;
- проверку события;
- нормализацию owner/referral;
- извлечение scores из массива `scores`;
- запись каждого score в отдельную колонку;
- запись `scores_json`;
- запись `syndromes_json`;
- запись ошибки в лог при неверном token.

## [ЧТО НЕ ОБРАБАТЫВАЕТСЯ]

Пока нет подтверждённого слоя:

- построения графика;
- правил шкалы графика;
- цветов зон;
- подписей к значениям;
- правил, какие точки считать критическими;
- правил порядка разбора;
- расчёта синдромов, если сайт их не присылает;
- генерации seller talk track по графику;
- генерации owner-facing interpretation;
- сохранения `graph_data_json`;
- сохранения `interpretation_json`;
- сохранения `top_risks`;
- сохранения `recommended_triggers`;
- формирования PDF/картинки результата.

## [ГРАФИК]

Из транскрибации разбора видно предположение:

- график может иметь диапазон от `-100` до `100`;
- значения ниже `0` интерпретируются как отсутствие/недостаток инструментов;
- разбор начинается с целей/первой точки и дальше идёт по показателям.

Но это пока не оформлено как подтверждённое техническое правило.

Нельзя внедрять автоматическую интерпретацию графика без решения Главного стратега.

## [ГЛАВНЫЙ РАЗРЫВ]

Сейчас система умеет:

```text
получить scores -> сохранить scores -> дать owner записаться на разбор
```

Но ещё не умеет подтверждённо:

```text
scores -> graph_data -> интерпретация -> сценарий разбора -> подсказки seller -> следующий шаг продажи
```

## [ЧТО НУЖНО ОТ ГЛАВНОГО СТРАТЕГА]

Нужно запросить отдельное задание:

```text
Этап: AS-BUSINESS-TEST-PROCESSING__01_GRAPH_AND_INTERPRETATION_RULES

Цель:
дать правила обработки результата бизнес-теста, чтобы Codex/n8n/сайт могли строить график, интерпретацию, подсказки для разбора и следующий шаг продажи курса.
```

## [ЗАПРОС ДАННЫХ СТРАТЕГУ]

Нужно получить:

1. Полный список показателей теста.
   - Подтвердить: показателей 9 или 10?
   - Если 10, дать точное название 10-го.

2. Для каждого показателя:
   - точное название в JSON;
   - точное название в Google Sheets;
   - порядок на графике;
   - минимальное значение;
   - максимальное значение;
   - что значит высокий балл;
   - что значит низкий балл;
   - какие вопросы/ответы формируют показатель.

3. Формулу расчёта:
   - как ответы теста переводятся в score;
   - где считается score: сайт / n8n / отдельный processor;
   - как обрабатывать пропущенные ответы;
   - округление;
   - допустимые диапазоны.

4. Правила графика:
   - диапазон оси;
   - цвета зон;
   - где ноль;
   - что показывать owner;
   - что показывать seller;
   - нужна ли картинка/PDF.

5. Правила интерпретации:
   - какие значения считаются критическими;
   - какие точки разбирать первыми;
   - как выбирать главную боль;
   - как определять сильные стороны;
   - как строить summary для owner;
   - как строить talk track для seller.

6. Правила синдромов:
   - список синдромов;
   - условия срабатывания;
   - текст для owner;
   - текст для seller;
   - связь с продажным триггером.

7. Правила продажи после разбора:
   - когда предлагать курс за `60 000 ₽`;
   - какой следующий шаг;
   - какие статусы продажи;
   - какие возражения фиксировать;
   - какие формулировки допустимы.

8. Примеры:
   - 3–5 реальных или тестовых результатов;
   - соответствующий график;
   - правильная интерпретация;
   - как seller должен провести разбор.

## [ПРЕДЛАГАЕМАЯ ТЕХНИЧЕСКАЯ АРХИТЕКТУРА]

После получения правил создать отдельный processor:


7. Сохранять данные в новую owner/razbor таблицу.
8. Передавать seller подсказки перед Zoom-разбором.

## [РИСКИ]

- Если график строить без правил Стратега, seller может неправильно интерпретировать результат.
- Если показателей всё-таки 10, текущий webhook теряет 10-й показатель.
- Если сайт уже считает график, нужно не дублировать расчёт в n8n без сверки.
- Если `scores_json` не содержит все ответы/сырые данные, восстановить расчёт будет невозможно.
- Если owner видит одно, а seller другое, будет конфликт на разборе.

## [ЗАПРЕТ]

- Не менять текущий production webhook без отдельного решения.
- Не утверждать 10-й показатель без данных.
- Не строить автоматическую интерпретацию как подтверждённую.
- Не менять названия колонок в живых таблицах без отдельного решения.
- Не показывать seller неподтверждённые триггеры как методологию.
