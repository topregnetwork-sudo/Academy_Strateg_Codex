## [КАК ОПРЕДЕЛЯЕТСЯ БЭТМАН]

### Входная команда

Узел `ROUTER — определить команду`:

- берёт текст из Telegram update;
- нормализует текст через `toLowerCase()` и схлопывание пробелов;
- распознаёт:
  - `бэтмен` / `бетмен` -> `batman`;
  - `команда` -> `team`;
  - `моя статистика` -> `my_stats`.

### Идентификация пользователя

Роутер формирует:

- `router_chat_id` из `chat.id` или `from.id`;
- `router_username` из username в нижнем регистре с `@`.

В ветках `Бэтмен`, `Команда`, `Моя статистика` кандидат ищется не по username, а по:

- `telegram_chat_id == router_chat_id`.

Если найдено несколько строк, берётся последняя по:

- `row_number` или `_rowNumber`.

### Активация Бэтмана

Узел `CODE — активировать Бэтмана`:

- ищет кандидата по `telegram_chat_id`;
- требует `статус_приглашения_в_TG_группу == отправлено`;
- если `btm_id` уже есть, использует его;
- если `btm_id` пустой, генерирует следующий `btm_XXXXXX` как `max(existing btm_id) + 1`.

Затем `UPDATE — записать btm_id` пишет `btm_id` в `Кандидаты_HR` по `candidate_id`.

## [КАК ВЫБИРАЕТСЯ ССЫЛКА]

### `business_test_main`

Узел `READ — Ссылки_Бэтманов business_test` читает весь лист `Ссылки_Бэтманов`.

Узел `CODE — найти business_test_main` ищет первую строку, где:

- `btm_id == btm_id_final`;
- `Тип ссылки == business_test_main`.

Есть fallback:

- `type == business_test_main`;
- URL берётся из `URL / формула`, fallback `URL`, fallback `url`.

### `hr_invite`

Узел `READ — Ссылки_Бэтманов hr_invite` читает весь лист `Ссылки_Бэтманов`.

Узел `CODE — найти hr_invite` ищет первую строку, где:

- `btm_id == btm_id_final`;
- `Тип ссылки == hr_invite`.

Есть fallback:

- `type == hr_invite`;
- URL берётся из `URL / формула`, fallback `URL`, fallback `url`.

### Что НЕ проверяется при выборе ссылки

При выборе ссылки workflow не проверяет:

- `Статус`;
- `Выдана`;
- `Дата выдачи`;
- `Нельзя менять после выдачи`.

Ссылка выбирается не как "свободная", а как "заранее привязанная к btm_id и типу".

## [КАК ПОМЕЧАЕТСЯ ВЫДАННОЙ]

### `business_test_main`

Сначала `UPDATE — выдать business_test_main кандидату` пишет в `Кандидаты_HR` по `candidate_id`:

- `btm_id`;
- `персональная_ссылка_бизнес_теста`;
- `статус_выдачи_бизнес_теста = выдано`.

Потом `UPDATE — пометить business_test_main выданной` обновляет лист `Ссылки_Бэтманов` по:

- match by `link_id`.

Пишет:

- `Статус = выдана`;
- `Выдана = да`;
- `Дата выдачи = now_iso` из активации Бэтмана.

Не пишет:

- `btm_id`.

### `hr_invite`

`UPDATE — пометить hr_invite выданной` обновляет лист `Ссылки_Бэтманов` по:

- match by `link_id`.

Пишет:

- `Статус = выдана`;
- `Выдана = да`;
- `Дата выдачи = new Date().toISOString()`.

Не пишет:

- `btm_id`;
- статус выдачи HR-ссылки в `Кандидаты_HR`.

## [ЗАЩИТА ОТ ДУБЛЕЙ]

### Что защищено

Повторная команда `Бэтмен` для уже активированного кандидата:

- не генерирует новый `btm_id`, если `btm_id` уже есть;
- снова найдёт ссылку `business_test_main` по тому же `btm_id`;
- с высокой вероятностью отправит ту же самую ссылку повторно.

Повторная команда `Команда`:

- требует уже существующий `btm_id`;
- снова найдёт `hr_invite` по тому же `btm_id`;
- с высокой вероятностью отправит ту же HR-ссылку повторно.

Одна и та же ссылка не должна попасть двум разным людям, если таблица `Ссылки_Бэтманов` заранее корректна: один `btm_id` -> одна строка `business_test_main` и одна строка `hr_invite`.

### Что НЕ защищено

Нет явной защиты на уровне workflow от:

- повторной отправки той же ссылки тому же `btm_id`;
- повторной перезаписи `Статус`, `Выдана`, `Дата выдачи`;
- дублей строк `btm_id + Тип ссылки` в `Ссылки_Бэтманов`;
- гонки при одновременной активации двух кандидатов без `btm_id`.

### Главный риск гонки

Если два пользователя одновременно выполнят `Бэтмен`, и у обоих ещё нет `btm_id`, оба запуска могут:

1. прочитать один и тот же текущий максимум `btm_id`;
2. сгенерировать одинаковый следующий `btm_id`;
3. записать одинаковый `btm_id` двум кандидатам.

После этого оба могут получить одну и ту же заранее привязанную ссылку.

## [СТАТИСТИКА]

Ветка `Моя статистика`:

1. Читает `Кандидаты_HR`.
2. Узел `CODE — найти btm_id для статистики` ищет кандидата по `telegram_chat_id`.
3. Требует существующий `btm_id`.
4. Читает лист `БизнесТест_Переходы`.
5. Узел `CODE — посчитать статистику` считает строки, где:
   - `btm_id == btm_id_final`;
   - `type == business_test_main`;
   - `статус == redirected`.

Итоговое сообщение показывает:

- `Бэтман ID`;
- количество переходов по бизнес-тесту.

## [РИСКИ]

1. Нет проверки `Статус`, `Выдана`, `Дата выдачи` перед выдачей ссылки.
2. Повторная команда повторно отправляет ту же ссылку и обновляет дату выдачи.
3. Нет защиты от дублей `btm_id + Тип ссылки` во внешней таблице.
4. Нет защиты от гонки генерации нового `btm_id`.
5. Статистика зависит от точных значений `type = business_test_main` и `статус = redirected`; другой регистр или другое написание не засчитаются.
6. Поиск кандидата завязан на `telegram_chat_id`; если он пустой или изменён, username не спасает поиск.
7. `READ — Ссылки_Бэтманов ...` читает весь лист и фильтрует в коде; при росте таблицы это может стать медленнее.
8. Fallback-поля `type`, `URL`, `url` на листе `Ссылки_Бэтманов` отсутствуют в подтверждённом снимке. Сейчас это не ломает логику, потому что есть `Тип ссылки` и `URL / формула`.
9. Не стоит чистить cached schema без отдельного решения: рабочая логика может зависеть от текущей структуры узлов n8n.

## [РЕШЕНИЕ / НЕ РЕШЕНИЕ]

### [ПОДТВЕРЖДЕНО]

Текущая логика работает как выдача заранее закреплённой ссылки по `btm_id`, а не как подбор первой свободной ссылки.

### [НЕ РЕШЕНИЕ]

Текущая логика не является полной защитой от дублей и гонок. Она не блокирует повторную выдачу той же ссылки тому же пользователю и не проверяет статус ссылки перед отправкой.

### [ГИПОТЕЗА]

Если таблица `Ссылки_Бэтманов` заранее генерируется как уникальная карта `btm_id + Тип ссылки`, текущая схема приемлема для MVP. Для масштабирования нужна явная проверка статусов и уникальности.

## [ОДИН СЛЕДУЮЩИЙ ШАГ]

Не менять workflow сразу. Сначала принять архитектурное решение: оставить модель "ссылка заранее закреплена за btm_id" или перейти к модели "выдавать первую свободную ссылку с блокировкой". После решения можно проектировать безопасную правку.
