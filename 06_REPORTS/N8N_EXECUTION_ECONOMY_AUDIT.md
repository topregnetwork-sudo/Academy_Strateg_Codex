# N8N_EXECUTION_ECONOMY_AUDIT

Документ для Стратега перед боевыми правками HR-Zoom / Batman / Owner-Razbor.

## [ПОДТВЕРЖДЕНО]

Проведён read-only аудит live n8n через API. Workflow не менялись, не активировались, не импортировались.

Локальный backup live workflow:

`03_N8N_WORKFLOWS/_LIVE_BACKUP_READONLY_AUDIT`

Сводка:

`03_N8N_WORKFLOWS/_LIVE_BACKUP_READONLY_AUDIT/20260711_091620_WORKFLOW_TRIGGER_SUMMARY.json`

По официальной модели n8n Cloud оплачиваются / лимитируются workflow executions, а не отдельные узлы. Один запуск workflow считается одним execution независимо от количества шагов внутри. Источник: https://n8n.io/pricing/

## [АКТИВНЫЕ WORKFLOW СЕЙЧАС]

| Workflow | Тип запуска | Активен | Роль |
|---|---:|---:|---|
| `AS-BOT__00_MAIN_ROUTER` | Telegram Trigger | да | главный Telegram-роутер HR/Batman-команд |
| `AS-BTM-OWNER-TEST-MVP__03_WEBHOOK_REGISTRATION_AND_RESULT` | Webhook | да | сайт → регистрация / результат бизнес-теста |
| `AS-HR-ZOOM-MVP__02_ZOOM_REMINDER_30MIN` | Schedule | да | напоминание за 30 минут |
| `AS-HR-ZOOM-MVP__03_ZOOM_CLICK_REDIRECT` | Webhook | да | фиксация Zoom-клика и redirect |
| `AS-HR-ZOOM-MVP__04_ZOOM_DOGON_10MIN` | Schedule | да | догонялка после начала Zoom |

Неактивные workflow не должны тратить executions, если их не запускать вручную. Но они создают риск путаницы при тестах и импортах.

## [КАК СЧИТАЮТСЯ КОМАНДЫ TELEGRAM]

Команды:

- `/start`;
- `вступить в группу`;
- `Бэтмен`;
- `Команда`;
- `Моя статистика`.

Они сейчас находятся внутри одного `AS-BOT__00_MAIN_ROUTER`. Это правильно: один Telegram Trigger принимает все входящие сообщения и внутри маршрутизирует команду.

Но каждый входящий Telegram update — это новый execution.

Пример для одного Бэтмана:

| Действие | Execution |
|---|---:|
| написал `/start` | 1 |
| написал `Бэтмен` | +1 |
| написал `Команда` | +1 |
| написал `Моя статистика` | +1 |

Итого: не один длинный процесс, а 4 отдельных запуска одного workflow.

Это нормально и экономичнее, чем держать отдельный Telegram Trigger под каждую команду. Отдельные Telegram Trigger на одного бота запрещены как архитектурный риск.

## [КАК СЧИТАЕТСЯ ZOOM CLICK]

`AS-HR-ZOOM-MVP__03_ZOOM_CLICK_REDIRECT` запускается каждый раз, когда кандидат нажимает Zoom-кнопку.

То есть:

| Действие | Execution |
|---|---:|
| кандидат нажал Zoom-кнопку | 1 |

Экономически это дополнительный расход. Но управленчески это важная метрика:

- кнопку дали;
- кандидат нажал;
- идентифицировали `candidate_id`;
- записали время клика;
- можно считать переходы;
- можно вывести время клика на `Доска_Прогресса`;
- можно отделить ранний клик от валидного.

Решение: Zoom-click webhook оставить. Его нельзя убирать, если доска прогресса должна показывать реальный клик и время перехода, а не просто зелёный круг.

## [ПОЧЕМУ НЕ ДЕЛАЕМ “КНОПКА АКТИВНА ТОЛЬКО ЗА 10 МИНУТ”]

Вариант без click webhook экономит executions, но ломает аналитику:

- Telegram не сообщает n8n о клике по обычной URL-кнопке;
- нельзя понять, кто нажал;
- нельзя показать точное время клика;
- нельзя отличить “увидел, но не нажал” от “нажал и не дошёл”;
- доска прогресса теряет управленческий смысл.

Для Академии Стратег важнее видеть поведение кандидата в воронке, чем сэкономить единичный execution на клик.

Решение: клики фиксируем.

## [ТЕКУЩАЯ ЭКОНОМИКА SCHEDULED WORKFLOW]

Сейчас `02_ZOOM_REMINDER_30MIN` запускается по слотам:

- Пн 7:30;
- Вт 7:30;
- Ср 7:30;
- Пн 17:30;
- Ср 17:30;
- Сб 5:30.

Это примерно 6 запусков в неделю, около 26 executions в месяц.

`04_ZOOM_DOGON_10MIN` запускается по слотам:

- Пн 8:10;
- Вт 8:10;
- Ср 8:10;
- Пн 18:10;
- Ср 18:10;
- Сб 6:10.

Это ещё примерно 6 запусков в неделю, около 26 executions в месяц.

Текущий scheduled расход HR-Zoom:

| Workflow | Примерно executions / месяц |
|---|---:|
| `02_ZOOM_REMINDER_30MIN` | 26 |
| `04_ZOOM_DOGON_10MIN` | 26 |
| Итого | 52 |

Это нормальный уровень. Опасность появится, если заменить точные schedule на “каждые 5 минут”: по оценке n8n, такой workflow даёт около 8600-8900 executions в месяц.

## [ГДЕ РЕАЛЬНАЯ ЭКОНОМИЯ]

Экономить нужно не на Zoom-click, а на количестве triggers и частоте schedule.

Нельзя:

- делать отдельный Telegram Trigger под каждую команду;
- делать scheduled workflow “каждые 5 минут” без крайней необходимости;
- плодить live workflow для тестовых версий;
- держать два активных Telegram Trigger на одного Telegram bot.

Можно:

- объединить `02 reminder`, будущий `05 host brief`, отправку Zoom-кнопки и `04 dogon` в один scheduled workflow;
- оставить schedule по точным слотам;
- отправлять host brief в том же запуске, что и reminder, если допустимо “за 30 минут”, а не строго “за 29 минут”;
- хранить import-ready версии локально, не активируя их в n8n.

## [ЦЕЛЕВАЯ ЭКОНОМНАЯ АРХИТЕКТУРА]

### Оставить отдельно

`AS-BOT__00_MAIN_ROUTER`

Назначение:

- `/start`;
- `вступить в группу`;
- `Бэтмен`;
- `Команда`;
- `Моя статистика`;
- в будущем можно добавить owner/seller router, если используется один Telegram bot.

Почему отдельно:

- Telegram messages всегда event-driven;
- один incoming message = один execution;
- объединение команд внутри одного router уже оптимально.

`AS-HR-ZOOM-MVP__03_ZOOM_CLICK_REDIRECT`

Назначение:

- фиксировать клик;
- писать `zoom_click_at_msk`;
- определять `early / valid / late`;
- redirect в Zoom;
- кормить `Доска_Прогресса`.

Почему отдельно:

- webhook должен мгновенно ответить кандидату;
- клик является отдельным пользовательским событием;
- это источник аналитики переходов.

`AS-BTM-OWNER-TEST-MVP__03_WEBHOOK_REGISTRATION_AND_RESULT`

Назначение:

- сайт → регистрация / результат бизнес-теста;
- owner_id / result_id / bot_link.

Почему отдельно:

- это сайт-вебхук, не Telegram и не HR schedule;
- критично быстро отвечать сайту.

### Объединить

Создать один workflow:

`AS-HR-ZOOM-MVP__SCHEDULED_ZOOM_EVENTS`

Внутри:

1. reminder за 30 минут;
2. host brief;
3. отправка / обновление Zoom-кнопки;
4. dogon после +10 минут;
5. summary ведущему после догона.

Запуск:

- по точным слотам, а не каждые 5 минут;
- лучше 2 группы расписаний:
  - `-30 min`: reminder + host brief + подготовка;
  - `+10 min`: dogon + dogon summary.

Экономика:

| Вариант | Примерно executions / месяц | Комментарий |
|---|---:|---|
| текущие `02 + 04` | 52 | уже приемлемо |
| добавить отдельный `05 host brief` | 78 | дороже, но всё ещё не критично |
| объединить `02 + 05`, оставить `04` | 52 | оптимально |
| один schedule каждые 5 минут | 8600-8900 | не делать |

## [ДОСКА ПРОГРЕССА]

Доска должна показывать не только цвет, но и время / счётчики:

- `last_zoom_message_id` — Zoom-кнопка / сообщение выдано;
- `last_zoom_datetime` — когда выдано;
- `zoom_click_at_msk` — последний raw click;
- `zoom_click_valid_at_msk` — валидный click в окне;
- `zoom_click_status` — `early / valid / late`;
- `zoom_click_early_count`;
- `zoom_click_late_count`;
- `статус_dogon_10мин`;
- `dogon_10min_sent_at`;
- `статус_Zoom` — отдельно, не равен факту клика.

Факт прихода в Zoom не считать по клику. Факт прихода подтверждается отдельно: кодовая фраза / ведущий / post-Zoom workflow.

## [РИСКИ]

1. Если убрать click webhook, потеряем аналитику переходов и время клика.
2. Если оставить прямую Zoom-ссылку текстом в сообщении, кандидат сможет обойти click webhook.
3. Если сделать schedule каждые 5 минут, можно сжечь месячный лимит executions.
4. Если создать отдельный Telegram Trigger под owner/seller/HR на того же бота, появится риск конфликтов и лишних executions.
5. Если держать много похожих live workflow, можно случайно активировать не ту версию.
6. Если dogon смотрит на legacy `zoom_click_at`, ранний клик будет ошибочно спасать кандидата от догона.

## [РЕШЕНИЕ ДЛЯ СТРАТЕГА]

Клики сохраняем.

Zoom-click webhook оставляем как источник аналитики:

- кто нажал;
- когда нажал;
- ранний / валидный / поздний клик;
- данные для доски прогресса;
- счётчик переходов.

Экономию делаем не через отказ от кликов, а через:

- один главный Telegram router;
- один scheduled HR-Zoom workflow по точным слотам;
- запрет на schedule каждые 5 минут;
- запрет на дубли active Telegram Trigger;
- локальное хранение import-ready workflow без активации.
